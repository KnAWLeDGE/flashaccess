FlashAccess Documentation

FlashAccess gives you a time-limited, IP-restricted web dashboard and MySQL credentials for a single session. When time runs out the temp user is dropped and the firewall rule is removed automatically.

Requirements

  • Ubuntu 24.04 (22.04 works; other Linux distros need manual UFW setup)
  • MySQL 8.0+ running locally (socket or TCP)
  • nginx installed (apt install nginx)
  • UFW enabled (ufw enable) — FlashAccess manages rules automatically
  • A domain pointed at your server, behind Cloudflare (Full Strict TLS)
  • A Cloudflare origin certificate for your domain
Note: FlashAccess itself does not open any public ports. It listens on 127.0.0.1:7432 and is exposed only through nginx. UFW rules it manages are for direct MySQL TCP access used by the CLI activate path (Navicat etc.).

Installation

Run the one-liner as root on your Ubuntu server:

$ curl -fsSL https://raw.githubusercontent.com/KnAWLeDGE/flashaccess/main/install.sh | bash

The installer will:

  1. Detect your architecture (amd64 / arm64) and download the correct binary to /usr/local/bin/flashaccess
  2. Create a flashaccess system user
  3. Create /var/lib/flashaccess/ (mode 0700) for encrypted config and session state
  4. Install the systemd unit at /etc/systemd/system/flashaccess.service
  5. Write a sample nginx config to /etc/nginx/sites-available/flashaccess (if nginx is installed)

Manual install

Download a binary directly from GitHub Releases:

$ wget https://github.com/KnAWLeDGE/flashaccess/releases/latest/download/flashaccess-linux-amd64
$ chmod +x flashaccess-linux-amd64
$ sudo mv flashaccess-linux-amd64 /usr/local/bin/flashaccess

flashaccess connect

Before FlashAccess can start a session it needs to know how to reach MySQL and needs an admin password to protect the session-creation UI. Run the wizard as root (or as the flashaccess system user with write access to /var/lib/flashaccess/): 

$ sudo flashaccess connect

The wizard prompts for:

  • MySQL host — default 127.0.0.1
  • MySQL port — default 3306
  • MySQL socket — leave blank for TCP; enter e.g. /var/run/mysqld/mysqld.sock for socket auth
  • MySQL user — typically root
  • MySQL password — empty if using socket auth
  • Dashboard listen address — default 127.0.0.1:7432
  • Admin password — minimum 8 characters; protects the session-creation form

The wizard verifies the MySQL connection before saving. Config is encrypted with AES-256-GCM and stored at /var/lib/flashaccess/config.enc. The master key is at /var/lib/flashaccess/master.key — back this up.

Warning: If you lose master.key, the config cannot be decrypted. Run flashaccess connect again to reconfigure.

Starting the server

Start FlashAccess manually to verify it works:

$ sudo flashaccess serve
FlashAccess dashboard listening on http://127.0.0.1:7432

Then enable and start it as a service (see systemd section):

$ sudo systemctl enable --now flashaccess
$ sudo systemctl status flashaccess

Sessions

A session is a time-limited access grant. It consists of a cryptographic key, an allowed IP/CIDR, a duration, and a temporary MySQL user provisioned for that session only.

Creating a session via the web dashboard

  1. Visit https://db.yourdomain.com
  2. Enter your admin password
  3. Fill in the allowed IP or CIDR (e.g. 203.0.113.42 or 10.0.0.0/24), session duration, and any security options
  4. Submit — the session key is shown once. Copy it now.
  5. You'll be redirected to the key gate. Paste the key to enter the dashboard.
Key shown once: The raw key is never stored. Only its Argon2id hash is persisted. If you lose the key, end the session and create a new one.

Session duration

Sessions accept Go duration strings: 30m, 2h, 12h, 24h. The maximum is 24 hours. The timer is shown in the dashboard topbar and colour-shifts to amber at 15 minutes and red at 5 minutes.

Ending a session

Sessions end automatically on expiry. To end early:

  • Click End Session in the dashboard topbar, or
  • Run flashaccess session end <id> from the server CLI

On end: the temporary MySQL user is dropped, the UFW rule is closed, and all web tokens are revoked.

Web dashboard

The web dashboard is accessible only from the session's allowed IP. It provides:

  • Database list — all databases visible to the admin connection
  • Table browser — engine, estimated row count, size; Browse and Structure links per table
  • Row browser — paginated table view with sortable columns (25 / 50 / 100 rows)
  • Structure view — full column info (type, nullable, key, default, extra, comment) and indexes
  • SQL editor — run arbitrary queries; Ctrl+Enter submits; results or affected-row count shown inline
Root access: The dashboard connects as your configured admin user (typically root). All SQL editor queries run with that privilege. There is no query filtering or sandbox.

CLI activate (Navicat / TablePlus / mysql)

If you prefer to use a GUI client (Navicat, TablePlus, DBeaver) or the mysql CLI directly, use the activate command. This verifies your session key and prints temporary credentials for the session's MySQL user: 

$ flashaccess session activate <session-id> <key> <your-ip>

GRANTED — session a91f3c2d7b40, 29m44s remaining
Connect with:
  host:     127.0.0.1:3306
  user:     tmp_a91f3c
  password: Kx9mRp2LnQv4

Use those credentials in Navicat etc. The TCP port is protected by UFW — only the allowed IP can connect.

Find your session ID: It's shown on the key-display page after session creation. You can also see active sessions via the dashboard URL path (e.g. /dashboard).

systemd service

The installer creates /etc/systemd/system/flashaccess.service. Key properties:

  • Runs as the dedicated flashaccess system user
  • Restarts automatically on failure (RestartSec=5s)
  • Hardened: NoNewPrivileges, ProtectSystem=strict, PrivateTmp
  • Listen address controlled by FLASHACCESS_ADDR env var (default 127.0.0.1:7432)
$ sudo systemctl enable --now flashaccess   # start + enable on boot
$ sudo systemctl status flashaccess
$ sudo journalctl -u flashaccess -f         # follow logs

nginx + Cloudflare

FlashAccess must run behind nginx with Cloudflare Full (Strict) TLS. This is required because IP enforcement reads CF-Connecting-IP — without Cloudflare, any client could spoof this header.

1. Install your origin certificate

In Cloudflare dashboard → SSL/TLS → Origin Server → Create Certificate. Save the certificate and key to your server:

/etc/ssl/cloudflare/origin.crt
/etc/ssl/cloudflare/origin.key
$ chmod 600 /etc/ssl/cloudflare/origin.key

2. Configure nginx

The installer writes a sample config to /etc/nginx/sites-available/flashaccess. Edit it to set your domain:

$ sudo nano /etc/nginx/sites-available/flashaccess
# Replace YOURDOMAIN.COM with your actual domain

Then enable it:

$ sudo ln -s /etc/nginx/sites-available/flashaccess /etc/nginx/sites-enabled/
$ sudo nginx -t           # verify config
$ sudo systemctl reload nginx

3. Set Cloudflare SSL mode

In Cloudflare dashboard → SSL/TLS → Overview: set mode to Full (Strict).

4. Lock nginx to Cloudflare IPs (optional but recommended)

The sample nginx config includes a geo block that sets a variable based on whether the request comes from a Cloudflare IP range. You can go further and add allow / deny directives to the server block so nginx itself refuses non-Cloudflare connections. Keep the IP list up to date from cloudflare.com/ips-v4.

UFW / firewall

FlashAccess calls ufw to open and close MySQL TCP access (port 3306 by default) for the session's allowed IP on session start and end. UFW must be installed and enabled. 

$ sudo ufw enable
$ sudo ufw status

To disable firewall management (e.g. on a dev machine without UFW): 

Environment=FLASHACCESS_NOFW=1   # add to the systemd unit's [Service] section

The dashboard web port (7432) should not be open in UFW — it only needs to be reachable from localhost, where nginx proxies to it.

Security model

  • AES-256-GCM — config file encrypted at rest with a per-machine master key
  • Argon2id — session keys hashed before storage; the raw key is never persisted
  • bcrypt — admin password hashed (cost 12); 500ms artificial delay on failed login
  • IP enforcement — checked on every authenticated request at the application layer via CF-Connecting-IP
  • Brute-force lockout — 5 failed key attempts locks the session for 5 minutes
  • HttpOnly / Secure / SameSite=Strict cookies — web tokens issued as secure cookies
  • In-memory token store — web tokens are not persisted; a server restart requires re-authentication
  • Temporary MySQL users — named tmp_<hexid>; validated by a strict regex before any SQL; dropped on session end
Threat model: FlashAccess is designed to protect against opportunistic access and credential leakage — not against a sophisticated attacker who has already compromised the server. The master key at /var/lib/flashaccess/master.key must be protected.

Environment variables

  • FLASHACCESS_ADDR — override the listen address (default 127.0.0.1:7432)
  • FLASHACCESS_NOFW — set to 1 to skip all UFW calls (dev / testing)

Building from source

Requires Go 1.23+.

$ git clone https://github.com/KnAWLeDGE/flashaccess.git
$ cd flashaccess
$ make build          # produces bin/flashaccess
$ make install        # copies to /usr/local/bin (needs sudo)

Cross-compile for both Linux targets:

$ make release        # produces dist/flashaccess_v*_linux_{amd64,arm64} + checksums.txt

Releasing a new version

  1. Commit all changes and ensure the working tree is clean
  2. Tag the release: git tag v1.0.0 && git push origin v1.0.0
  3. Run make release — outputs dist/flashaccess_v1.0.0_linux_amd64, _arm64, and checksums.txt
  4. Create a GitHub Release on the tag; upload the dist/ files as release assets
  5. The install script resolves the latest tag automatically via the GitHub API — no changes needed

Build flags inject the version at link time via -X main.version=$(VERSION) (see makefile).
